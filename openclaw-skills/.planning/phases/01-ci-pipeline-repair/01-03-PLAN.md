    1. CHANGE `runs-on: macos-14` to `runs-on: macos-15`

    3. In the `Build for simulator` step, CHANGE:

    Do NOT modify the lint-kotlin, lint-swift, skills-test, or android-build jobs in ci.yml.

    --- Changes to ios/OpenClawConsole/fastlane/Fastfile ---

    Replace the entire contents with:

    ```ruby
    default_platform(:ios)

    platform :ios do
      before_all do
        setup_ci if ENV['CI']
      end

      desc "Push a new beta build to TestFlight"
      lane :beta do
        build_number = ENV['BUILD_NUMBER'] || ENV['GITHUB_RUN_NUMBER'] || (latest_testflight_build_number + 1).to_s
        increment_build_number(build_number: build_number)
        build_app(scheme: "OpenClawConsole")
        upload_to_testflight
      end

      desc "Setup certificates and profiles via match"
      lane :setup do
        match(type: "appstore")
      end

      desc "Distribute to Firebase App Distribution"
      lane :firebase_dev do
        build_number = ENV['BUILD_NUMBER'] || ENV['GITHUB_RUN_NUMBER'] || (latest_testflight_build_number + 1).to_s
        increment_build_number(build_number: build_number)
        build_app(
          scheme: "OpenClawConsole",
          export_method: "ad-hoc"
        )
        firebase_app_distribution(
          app: ENV["FIREBASE_IOS_APP_ID"],
          groups: "qa-team,developers",
          release_notes: "Dev build from develop branch (#{ENV['GITHUB_SHA'] || 'local'})"
        )
      end
    end
    ```

    Key changes vs original Fastfile:
    - Added `before_all do; setup_ci if ENV['CI']; end` — prevents keychain hang on headless CI (CI-04)
    - `:beta` lane: `increment_build_number` now reads ENV['BUILD_NUMBER'] first (injected by workflow as github.run_number), falling back to ENV['GITHUB_RUN_NUMBER'], then lastly to latest_testflight_build_number+1 (CI-06)
    - `:firebase_dev` lane: same build_number pattern applied for consistency
    - All other lane logic preserved exactly (scheme, export_method, upload_to_testflight, firebase_app_distribution params)

    Note on .xcodeproj: The iOS directory uses Package.swift (SPM package), not a .xcodeproj.
    The `build_app(scheme: "OpenClawConsole")` call requires an Xcode project file to work in Fastlane.
    This is a known open question from the research. The Fastfile changes here are necessary regardless —
    they fix CI-04 and CI-06 correctness. Whether `fastlane beta` can build without a .xcodeproj is a separate
    structural issue outside Phase 1 scope. The internal-distribution.yml ios-testflight-internal job will
    fail at `fastlane beta` if no .xcodeproj exists, but that is a Phase 2 concern (code signing). Phase 1
    only requires the CI syntax/configuration to be correct.
